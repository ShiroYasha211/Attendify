# دليل نظام الإدارة (Admin Module) - Attendify

هذا الملف يوثق جميع الجوانب التقنية والوظيفية للوحة تحكم الإدارة (Admin Panel) في نظام **Attendify**، متضمناً واجهات الويب (Web Views) وواجهات برمجة التطبيقات (API).

---

## 1. نظرة عامة (Overview)

نظام الإدارة هو القلب النابض لـ Attendify، يوفر تحكماً كاملاً للجامعة/الكلية في إدارة الهيكلية الأكاديمية (جامعات، كليات، تخصصات، مواد)، المستخدمين (طلاب، دكاترة، مناديب)، إلى جانب استعراض التقارير الدقيقة عن الحضور والغياب، وإدارة الإعدادات العامة وسجل النشاطات.

### التقنيات الأساسية

- **Backend:** Laravel 10+
- **Frontend (Web):** Blade Templates مع CSS نقي (Vanila CSS) ومكونات متجاوبة (Responsive).
- **API Authentication:** Laravel Sanctum (Bearer Tokens).
- **Role Management:** نظام أدوار مخصص (`UserRole` Enum).

---

## 2. الهيكلية المعمارية (Architecture)

يتكون نظام الإدارة من مسارين رئيسيين يشتركان في نفس قاعدة البيانات والمنطق الأساسي (Business Logic):

### أ. واجهة الويب (Web Interface)

- **المسار (Route Prefix):** `/admin`
- **الميدل وير (Middleware):** `auth`, `status` (للتأكد من أن الحساب نشط)، وفحص الصلاحية كمدير `UserRole::ADMIN`.
- **موقع الكنترولات:** `app/Http/Controllers/Admin/`
- **موقع العروض (Views):** `resources/views/admin/`

### ب. واجهة التطبيقات (REST API)

- **المسار (Route Prefix):** `/api/admin`
- **الميدل وير (Middleware):** `auth:sanctum`، وفحص الصلاحية.
- **موقع الكنترولات:** `app/Http/Controllers/Api/Admin/`
- **هيكلية الرد (Response Structure):**
  جميع الردود تتبع معياراً واحداً لتسهيل استهلاكها من تطبيق Flutter:

  ```json
  {
      "success": true,
      "message": "رسالة توضيحية",
      "data": { ... } // أو مصفوفة بيانات
  }
  ```

## 3. الوحدات الأساسية (Core Modules)

### 3.1. لوحة المعلومات (Dashboard)

- **الوظيفة:** توفير نظرة عامة سريعة للإدارة.
- **البيانات المعروضة:**
  - إحصائيات سريعة (أعداد الطلاب، الدكاترة، المواد، إلخ).
  - نسبة الحضور والغياب العامة واليومية.
  - الطلاب المنذرين (At-Risk Students) الذين تجاوزوا نسبة الغياب المسموحة.
  - مسار الحضور خلال آخر 7 أيام (7-Day Trend Chart).
  - أكثر المواد تسجيلاً للغياب.

### 3.2. الإدارة الأكاديمية (Academic Management)

يتم إدارة الهيكلية الأكاديمية بشكل هرمي متسلسل:

1. **الجامعات (Universities):** الكيان الأعلى.
2. **الكليات (Colleges):** تتبع لجامعة معينة.
3. **التخصصات (Majors):** تتبع لكلية معينة.
   - **ميزة الأتمتة:** عند إنشاء تخصص، يطلب النظام عدد المستويات (السنوات) وعدد الفصول (الأتْرام) في كل مستوى، ويقوم `AcademicService` بإنشاء الهيكلية كاملة تلقائياً (مثال: إنشاء مستوى 1 -> ترم 1، ترم 2).
   - **التخصصات السريرية (Clinical):** يمكن تعليم التخصص كـ "سريري" لتمكين إضافة "مندوب سريري" لاحقاً للتحضير اليدوي/التقييم.
4. **المواد (Subjects):** تتبع لفصل دراسي (Term) معين، ويتم ربطها بدكتور (Doctor).
   - **الغياب:** يتم تحديد الحد الأقصى للغياب لكل مادة.

### 3.3. إدارة المستخدمين (User Management)

- **الأنواع (Roles):** مدراء (Admin)، دكاترة (Doctor)، طلاب (Student)، مناديب (Delegate).
- **العمليات المتوفرة:**
  - عرض القوائم مع الفلترة والبحث.
  - إيقاف/تفعيل الحسابات (يمنع المستخدم الموقوف من الدخول للنظام).
  - إضافة وتعديل وحذف (Soft Delete) للمستخدمين.
  - **الإجراءات الجماعية (Bulk Actions):** تفعيل، تعطيل، أو حذف مجموعة من المستخدمين دفعة واحدة بناءً على التحديد.
  - **الربط الأكاديمي:**
    - الطالب يُربط بـ (المستوى -> التخصص -> الكلية -> الجامعة) أوتوماتيكياً عند اختيار المستوى.
    - المندوب يُربط بنفس الطريقة.

### 3.4. المناديب السريرية (Clinical Delegates)

- **الوظيفة:** تعيين طالب كمندوب لمتابعة الحضور السريري لتخصص كامل.
- **الآلية:** يتم عرض التخصصات التي تم تعليمها كـ `has_clinical=true` فقط، ويمكن للإدارة اختيار طالب من نفس التخصص ليكون المندوب السريري.

### 3.5. التقارير (Reports)

- **الوظيفة:** استخراج بيانات دقيقة عن الحضور والغياب للمتابعة واتخاذ القرارات.
- **أنواع التقارير:**
  1. **نظرة عامة (Overview):** إحصائيات عامة للحضور.
  2. **تقرير المادة (Subject Report):** عرض حضور وغياب كل طالب لمادة معينة بشكل تفصيلي.
  3. **تقرير الحرمان (Threshold/Deprivation):** حساب نسبة الغياب للطلاب في مستوى معين بناءً على عتبة (Threshold) بالمئة يحددها المدير. (مثال: من تجاوز 25% من غياب أي مادة).
  4. **أداء الدكاترة (Doctor Performance):** عرض معدلات الحضور للمواد المرتبطة بكل دكتور.
  5. **نظرة شاملة للنظام (System Overview):** موجز كامل للنظام.

### 3.6. الإعدادات وسجل النشاطات (Settings & Activity Logs)

- **الإعدادات (Settings):** إدارة القيم الثابتة والمتغيرة للنظام (مثل اسم النظام، أوقات السماحية، إلخ) وتحديثها مركزياً. يتم مسح الـ Cache تلقائياً عند التعديل.
- **سجل النشاطات (Activity Log):**
  - تتبع دقيق (Audit Trail) لكل عملية مهمة في النظام (إضافة، تعديل، حذف، تغيير حالة حساب).
  - يتم استخدام `LogsActivity` Trait في الكنترولات لتسجيل الحدث مع معرف المستخدم والوصف.
  - إمكانية فلترة السجلات وحذف السجلات القديمة (التنظيف - Cleanup).

## 4. مسارات API (API Routes Directory)

جميع المسارات تبدأ بـ `[GET/POST/PUT/DELETE] /api/admin/`

| القسم | Endpoint الأبرز | الوصف |
|---|---|---|
| **المصادقة** | `/login` , `/logout` , `/me` | الدخول، الخروج، وملف المدير الشخصي |
| **الرئيسية** | `/dashboard` | إحصاءات مبسطة وتخاطط بيانية |
| **أكاديمي** | `/universities` , `/colleges` , `/majors` , `/subjects` | عمليات CRUD كاملة للهيكلية التعليمة |
| **المستخدمين** | `/users` , `/users/{id}/status` , `/users/bulk-*` | إدارة شاملة للمستخدمين بصلاحية عامة |
| **الأدوار** | `/students` , `/doctors` , `/delegates` | إدارة المتسخدمين بشكل مخصص لكل نوع |
| **السريري** | `/clinical-delegates` | تعيين وإزالة مناديب التدريب |
| **التقارير** | `/reports/overview` , `/reports/subject` , `/reports/threshold`... | سحب تقارير مفصلة لاتخاذ القرارات |
| **النظام** | `/settings` , `/activities` | إعدادات ومراقبة النظام |

## 5. الاعتبارات الأمنية (Security Considerations)

- **Sanctum Tokens:** لا يتم استخدام Session في الـ API، بل يعتمد بالكامل على Bearer Tokens تصدر عند تسجيل الدخول السليم وتلغى عند تسجيل الخروج.
- **Soft Deletes:** لمنع فقدان البيانات المرتبطة (مثل الحضور)، يتم استخدام Soft Deletes للمستخدمين والمواد.
- **Authorization:** كل كنترولات الأدمن سواء Web أو API محمية بطبقة تتأكد أن Role = Admin.

تم اعداد النظام بشكل يسمح لفريق Flutter بالبدء بالعمل المستقل والكامل لبناء تطبيق لوحة التحكم للأدمن.
